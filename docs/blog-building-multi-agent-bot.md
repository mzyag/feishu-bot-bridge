# 我在飞书里养了一支 AI 开发团队

## 起因

上周三早高峰，13 号线，我单手扶着杆，突然想起昨晚那个 PR 里有个边界条件漏了。掏出手机，在飞书里打了一行字："帮我在 validate_input 里加个空字符串判断"。三十秒后收到回复——代码改好了，测试通过。

这就是我现在每天通勤的状态。

我每天路上大概一个小时。以前想到什么只能记在备忘录里，到公司再处理——但说实话十次有八次到了就忘。SSH + tmux 在手机上试过，我只能说：如果你在晃动的地铁上用虚拟键盘敲过 `kubectl get pods --namespace=production`，你就知道那不是给人用的。

我真正想要的很简单：**用人话说一句，让 AI 把事干了**。

飞书和微信是我手机上打开频率最高的两个 App。如果能在这俩里面直接跟 Claude Code 对话，让它替我改代码、跑测试——那每天这一小时就不算浪费了。

## 第一版：先跑通再说

最开始我只想做一件事：飞书发消息 → Mac 上的 Claude Code CLI 执行 → 回复结果。

飞书这边好办，`lark-oapi` 有 WebSocket 长连接模式，不需要公网回调地址，本地就能跑。问题出在怎么调 Claude Code CLI。

我一开始用的是最直觉的方式：每条消息 spawn 一个 `claude -p "prompt"` 子进程，等执行完拿输出。能跑，但体验很烂——每次冷启动要 5-10 秒（它要加载 MCP servers、扫 CLAUDE.md），而且上下文不连续，每条消息都是孤立的。发完"帮我看下这个函数"再问"那个变量是干嘛的"，它已经忘了你在说什么了。

然后我翻 `claude --help` 翻到一个参数：`--input-format stream-json`。这个模式下 CLI 启动一次就不退出了，你往 stdin 写 JSON，它从 stdout 流式吐事件。上下文自动连续，因为压根就是同一个进程。

切到这个模式之后响应速度从 5-10 秒降到了 2-3 秒。体感是"发完消息刚把手机揣兜里它就回了"。质变。

## 真正的麻烦：launchd

我不想每次开机手动 `python3 ws_bot.py`，所以做成了 launchd 后台服务。

结果这一步踩的坑比写 bot 本身还多。

launchd 启动的进程几乎什么环境变量都没有。我在终端里跑得好好的代码，放到 launchd 里就开始花式报错——403、command not found、permission denied 轮着来。

让我最崩溃的一个问题排查了将近一小时：Claude CLI 启动了（日志里 session_id 都打出来了），API 调用也发了，但永远返回 403。我翻了四遍代码确认 token 没问题，去 Claude 官网手动试了 API key 能用，甚至怀疑是不是 Claude 把我封了……

一个小时后我才意识到：压根不是认证问题。**请求没走代理。** 我在国内，Claude API 要翻墙。终端里的 `http_proxy` 是 shell profile 设的，launchd 里根本没有这个变量。请求直连被墙了返回的也是 403。

我盯着屏幕愣了三十秒。一个小时啊。

类似的问题一连串：
- `HOME` 没设 → Claude 找不到 `~/.claude/` 配置
- `PATH` 里没有 npm global bin → `claude` 命令不存在
- OAuth token 在 Keychain 里 → launchd 有时能读有时不能（看心情，取决于 GUI session 状态）

最终我的 launchd plist 里手动加了 7 个环境变量才稳定下来。教训就一条：**任何依赖用户环境的 CLI 工具，放到后台服务都要把环境变量全部显式声明。** 不要假设后台进程能继承你终端里的任何东西。

## 一个反直觉的发现：exit code 不能信

Claude CLI 有个行为困惑了我很久：明明回复了正确答案，进程退出码却是 1。

追了半天发现是 MCP server 连接失败——比如某个插件超时了，Claude 就判定"出错了"所以 exit 1。但用户的问题它其实已经正确回答了。

我的做法是：**不看 exit code，只看 stdout 事件流**。流里有 `{"type": "result", "subtype": "success"}` 就算成功，不管 exit code 是啥。只有完全没输出的时候才走错误处理。

这个违反了我十几年写 shell 脚本的直觉，但确实是对的。

## 加微信：意外地顺利

飞书跑通之后我把微信也接了。用的是 openclaw 的微信 IM Bot 插件，协议很简单：

- `notifyStart` 注册
- `getUpdates` 长轮询（服务端 hold 住直到有新消息，跟 Telegram Bot API 一个思路）
- `sendMessage` 发消息

200 行代码就通了。关键设计：飞书和微信共享同一个 Claude 持久进程和消息队列——两个通道只是"收消息"和"发回复"的方式不同，中间逻辑完全一样。

有个小坑差点栽进去：微信 API 的域名（ilinkai.weixin.qq.com）是国内服务器，不用代理。但我全局配了 `all_proxy`，httpx 对所有请求都走 SOCKS。结果微信 API 报错 `socksio package not installed`——它试图用 SOCKS 代理访问一个本来就能直连的国内地址。`NO_PROXY` 里加上微信域名就好了。

## 从单次对话到多 Agent 团队

单条消息跑顺之后，我开始想：能不能处理更大的任务？比如"帮我把这个 2000 行的文件拆成多个模块"。

这种事一条消息搞不定。需要拆解、分步执行、验证、审查。手动来大概是：理解需求 → 出方案 → 确认 → 写代码 → 测试 → review → 交付。

我就把这套流程自动化了，用两个模型分工。

**Claude Opus** 负责动脑子的部分——需求分析、计划制定、最终审查。它的强项是理解复杂意图，能做"这个需求到底要改哪些文件、有什么风险"这类工程决策。

**DeepSeek V4 Pro** 负责动手的部分——写代码方案。它写得又快又好（尤其是算法和设计模式），3-5 秒出一个完整模块方案，成本还极低。

协作方式：DeepSeek 先出方案，Claude Code 拿着方案在真实项目环境里执行。因为 Claude 能读现有文件、看依赖关系、跑命令验证，所以它会根据实际情况对方案做调整——DeepSeek 写的代码有时候 import 路径是编的，Claude 在执行时会自动修正。

简单说：DeepSeek 出初稿快且便宜，Claude 落地准且能兜底。单用任何一个都不如两个配合。代价是链路复杂了一倍，任何一个挂了整条链就断。

简单任务（查状态、改配置）走纯 Claude 就够了，复杂任务才启动团队模式。Router 通过关键词自动判断，90% 的消息根本不需要调 LLM 做路由。

## TDD 驱动的执行流程

团队模式一开始的问题是：Executor 写完代码不知道对不对。要到最后 Reviewer 审查才发现方向错了，然后整个推翻重来。等于白跑一遍。

后来改成 TDD：Dispatcher 制定计划时，每一步除了写"做什么"，还要写**验收标准**（一个 assert）。执行变成了：

1. 先根据验收标准生成测试脚本（此时代码还不存在）
2. DeepSeek 出方案
3. Claude Code 执行（目标：让测试通过）
4. 跑测试
5. 通过 → 下一步；失败 → 修一轮再说

好处是 Executor 写代码时已经知道"什么算完成"。不会跑偏，也不会过度设计。

## 一次经典翻车

团队模式刚上线的第二天，我在地铁上发了句"帮我加个用户认证"。

Dispatcher 的理解是：给飞书 bot 加认证中间件。它规划了 5 个步骤，生成了完整的 JWT 验证方案，DeepSeek 出了代码，Claude 开始执行……

我其实想说的是：给我另一个项目的 API 加个简单的 token 校验。

等我从地铁出来看手机，bot 已经把 `ws_bot.py` 改得面目全非了——加了 JWT 库依赖、一堆 middleware、还新建了个 `auth/` 目录。代码质量不错，但方向完全不对。我花了十分钟 `git checkout .` 才恢复现场。

这件事之后我加了两个人工卡点：

1. **需求确认** — Dispatcher 分析完先告诉我"我理解你要做 X，涉及文件 Y，风险 Z"，我说"确认"才继续
2. **计划确认** — 执行计划列出来后，我看一遍确认合理再开始

执行过程中不需要人工介入——每步自动验证，有问题自动修一轮，修不了标记失败跳过。人的注意力应该花在"方向对不对"，不是"执行对不对"。

## 进程保活：Watchdog

持久进程最怕的状态是"活着但不干活"——`ps aux` 能看到进程，`poll()` 返回 None，但 stdout 死活不再输出了。

我遇过好几次：团队模式跑到第三步，突然没动静了。进程还活着，就是不出结果。手动 kill 重启就好。

加了个 Watchdog 线程：每 10 秒检查 `last_stdout_timestamp`，超过 90 秒没任何输出就判定僵死，直接 kill。下次 `send_message` 发现进程死了会自动重启（带 backoff 防快速循环）。

上线之后有一天早上起来看日志，发现凌晨 3 点 watchdog 连续重启了 4 次——Claude 那边疑似在做维护。但因为有 backoff（1s → 2s → 4s → 8s），没有打爆任何东西，等 Claude 恢复了自动就好了。如果没有 watchdog，第二天早上我就得手动重启。

## 几个交了学费的细节

**确认关键词不能用模糊匹配。** 我最初用 `"好" in text` 判断用户确认。某天我回了句"不好意思我在忙"——bot 看到了"好"，认为我确认了，直接开始执行。等我回过神来代码已经改了三个文件了。改成 `text.strip() == "确认"` 精确匹配后再没出过事。

**Workflow 状态必须持久化。** 用户发了需求，AI 回了"请确认"。用户五分钟后才回"确认"——但这五分钟里 bot 可能因为某个异常重启了，内存里的状态全没了。所以 workflow state 必须写文件，重启后能接上。

**相同任务要去重。** 网络不好的时候用户会重复发消息。执行前先查历史 PRD 文件，完成过的相同任务直接返回上次结果。

**LLM 输出格式永远不可信。** 哪怕 prompt 里用粗体加大字写了"只输出 JSON，不要其他任何文字"，Claude 还是时不时加一句"好的，以下是 JSON 输出："然后包个 markdown 代码块。必须用正则从整段文本里提取 JSON 对象，不能假设 response 就是纯 JSON。这个坑我踩了至少三次才学乖。

## 回头看

整个项目从 0 到能用花了大概 6 个小时。最终 11 个文件约 2000 行 Python，飞书 + 微信双通道。

如果让我重来——

我不会先做 per-message spawn 版本。当时觉得"先跑通最简单的再优化"，结果花两个小时调通之后就不想动了。"能跑"是优化的敌人。拖了两天才下决心重写成持久进程，等于那两天的使用体验白差了。

我也不会一上来就做团队模式。应该先把 single mode 打磨到无 bug，再加复杂度。多 Agent 的调试成本是指数级的——状态机、超时、确认流程、模型调度，随便一个环节出问题都难定位。

还有就是日常应该用 Sonnet 而不是 Opus。Opus 质量高但太慢，团队模式 20 次调用全走 Opus 要等好几分钟。后来改成"规划和审查用 Opus，执行用 Sonnet"之后体感速度翻倍。

测试该从第一天就写。我是攒到 2000 行才开始补的，早加测试后面能省很多排查时间。这条道理人人都知道，但写 side project 的时候谁不是"先跑通再说"呢。

## 现在

- 日常通勤用飞书/微信发简单任务（改 bug、查代码、跑命令），2-5 秒回复
- 复杂任务用团队模式，3-10 分钟自动交付，手机上只需要确认两次
- Watchdog 保活 + 自动重启，一周了没手动干预过
- 成本：Claude Max 订阅 + DeepSeek API（一天不到一块钱）

最满意的时刻不是技术实现多优雅，而是某天在地铁上发了句"帮我把 ws_bot.py 拆成多个模块"，到站的时候收到通知："重构完成，8 个文件全部测试通过。"走进办公室，`git pull`，代码已经整整齐齐摆在那了。

那种感觉就像——你有了一个随叫随到的搭档，而它永远不会说"等我到工位再看"。
