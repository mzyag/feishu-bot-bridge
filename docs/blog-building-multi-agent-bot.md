# 从飞书 Bot 到多 Agent 开发团队：用手机远程操控 Claude Code 的全过程

## 背景

我每天通勤路上大概一个小时，经常在地铁上想到代码要改什么。之前的做法是到公司再处理，但有些东西想到了不马上记下来就忘了。

SSH + tmux 我试过，手机上敲命令行实在太反人类——屏幕小、没有 Tab 补全、一个 typo 就要重来。我需要的是：**用自然语言发一句话，让 AI 帮我把事干了**。

飞书和微信是我手机上打开频率最高的 App。如果能在这两个里面直接跟 Claude Code 对话，让它替我读文件、改代码、跑测试，那通勤时间就不算浪费了。

## 第一版：先跑通最简单的路径

最开始我只是想做一件事：飞书发消息 → Mac 上的 Claude Code CLI 执行 → 回复结果。

飞书这边好办，`lark-oapi` 有 WebSocket 长连接模式，不需要公网回调地址，本地就能跑。问题出在怎么调 Claude Code CLI。

我一开始用的是最直觉的方式：每条消息 spawn 一个 `claude -p "prompt"` 子进程，等它执行完拿输出。能跑，但体验很差——每次冷启动要 5-10 秒（它要加载 MCP servers、扫 CLAUDE.md），而且上下文不连续，每条消息都是独立的。

翻了一下 `claude --help`，发现有个 `--input-format stream-json` 模式。这个模式下 Claude CLI 启动一次就不退出了，你往 stdin 写 JSON 消息，它从 stdout 流式输出事件。上下文自动保持，因为就是同一个进程在跑。

切到这个模式之后响应速度从 5-10 秒降到了 2-3 秒。这个提升是质变级别的。

## 真正的麻烦：launchd

我不想每次开机都手动 `python3 ws_bot.py`，所以做成了 launchd 服务。结果这一步踩了这个项目里最多的坑。

launchd 启动的进程几乎什么环境变量都没有。我在终端里跑得好好的代码，放到 launchd 里就各种 403、command not found。

最离谱的一个问题排查了快一个小时：Claude CLI 启动成功了（system.init 事件都打出来了），API 调用也发出去了，但返回 403。最后发现是**没有配代理**——我在国内，Claude API 需要走代理，终端里的 `http_proxy` 是 shell profile 设的，launchd 根本没有这个变量。

类似的问题一连串：
- `HOME` 没设 → Claude 找不到 `~/.claude/` 配置
- `PATH` 里没有 npm global bin → `claude` 命令不存在
- OAuth token 在 Keychain 里 → launchd 有时能读有时不能（取决于 GUI session 状态）

最终我的 launchd plist 里手动加了 7 个环境变量才稳定下来。教训很深刻：**任何依赖用户环境的 CLI 工具，放到后台服务里都要把环境变量全部显式声明。**

## 一个反直觉的发现：exit code 不能信

Claude CLI 有个行为让我困惑了很久：明明回复了正确答案，进程退出码却是 1。

追了半天发现是 MCP server 连接失败导致的——比如某个 MCP 插件超时了，Claude 认为这是个"错误"所以 exit 1，但实际上用户的问题它已经正确回答了。

我的做法是：**不看 exit code，看 stdout 的事件流**。只要流里面有 `{"type": "result", "subtype": "success"}` 事件，不管 exit code 是啥都算成功。只有完全没有输出的时候才走错误处理。

## 加微信：比想象中简单

飞书跑通之后，我把微信也接了进来。我之前用过一个叫 openclaw 的工具，它有个微信 IM Bot 插件，协议是公开的：

- `notifyStart` 注册
- `getUpdates` 长轮询（服务端 hold 住直到有新消息，跟 Telegram Bot API 类似）
- `sendMessage` 发消息

写了 200 行代码就接通了。关键设计是飞书和微信共享同一个 Claude 持久进程和消息队列——两个通道只是"收消息"和"发回复"的方式不同，中间的处理逻辑完全一样。

有个细节差点坑到我：微信 API 的域名（ilinkai.weixin.qq.com）是国内服务器，不需要代理。但我全局配了 `all_proxy`，httpx 会对所有请求都走 SOCKS 代理。结果微信 API 调用报错 `socksio package not installed`。解决方法是在 `NO_PROXY` 里加上微信的域名。

## 从单次对话到多 Agent 团队

单条消息的场景跑顺了之后，我开始想：能不能处理更复杂的任务？比如"帮我把这个 2000 行的文件拆成多个模块"。

这种任务一条消息搞不定，需要拆解、分步执行、验证、审查。人工操作的话大概是这个流程：理解需求 → 出方案 → 确认 → 写代码 → 测试 → review → 交付。

我就把这个流程自动化了，用两个模型分工：

**Claude Opus** 做需求分析、计划制定、最终审查——它的强项是理解复杂意图和多步推理，能做"这个需求到底要改哪些文件、有什么风险"这类工程决策。

**DeepSeek V4 Pro** 做代码方案生成——它写代码又快又好（尤其是算法实现和设计模式），成本还低。3-5 秒就能出一个完整模块的方案。

两个模型的协作方式：DeepSeek 先出一版代码方案，Claude Code 拿着这个方案在真实项目环境里执行——它能读现有文件、理解依赖关系、跑命令验证，所以会根据实际情况对方案做调整或修正。

| 维度 | 纯 Claude | 纯 DeepSeek | 双模型协作 |
| --- | --- | --- | --- |
| 代码生成质量 | 良好 | **优秀**(算法/模式) | 优秀 |
| 需求理解/决策 | **优秀** | 一般 | 优秀 |
| 速度 | 慢(10-30s/步) | **快**(3-5s/步) | 中等(并行可优化) |
| 成本 | 高 | **极低** | 中等 |
| 上下文感知 | **强**(读项目文件) | 弱(只看prompt) | 强 |
| 自主纠错 | **强** | 弱 | 强 |
| 链路复杂度 | 简单 | 简单 | **复杂**(双模型调度) |
| 单点故障 | 低 | 低 | **较高**(任一挂都影响) |

简单任务（查个状态、改个配置）走纯 Claude 就够了，复杂任务才启动团队模式。Router 通过关键词自动判断，90% 的消息不需要调 LLM 做路由。

## TDD 驱动的执行流程

团队模式一开始的问题是：Executor 写完代码不知道对不对，要到最后 Reviewer 审查才发现问题，然后整个推翻重来。

后来我改成了 TDD 的方式。Dispatcher 制定计划时，每一步不只写"做什么"，还要写**验收标准**（一个 assert 断言）。执行时的流程变成：

1. 先根据验收标准生成测试脚本（此时代码还不存在）
2. DeepSeek 出方案
3. Claude Code 执行（目标：让测试通过）
4. 跑预生成的测试
5. 通过 → 下一步；失败 → 修复一轮

这样做的好处是 Executor 写代码时**已经知道通过标准是什么**。不会写着写着跑偏，也不会过度设计。

## 进程保活：Watchdog 的必要性

持久进程最怕的状态是"活着但不干活"——`ps aux` 能看到进程，`poll()` 返回 None，但 stdout 已经不再有任何输出了。

我遇到过好几次：团队模式跑到第三步，突然没反应了。进程还在，但就是不出结果。手动 kill 重启就好了。

解法是加了个 Watchdog 线程：每 10 秒检查一次 `last_stdout_timestamp`，如果超过 90 秒没有任何 stdout 输出，判定为僵死，直接 kill 掉。下次 `send_message` 发现进程死了会自动重启（带 backoff 防止快速循环重启）。

## 人工确认放在哪

一开始我让整个流程全自动跑（发完需求直接执行到底）。结果经常出现：AI 理解的需求和我想的不一样，代码改了一通，方向完全错误。

后来加了两个人工卡点：
1. **需求确认** — Dispatcher 分析完后，先告诉我"我理解你要做 X，涉及文件 Y，风险 Z"，我确认了才往下走
2. **计划确认** — 执行计划列出来后，我看一遍，确认每一步合理再开始执行

执行过程中不需要人工确认——每步自动验证，有问题自动修复一轮，修不了就标记失败继续下一步。人的注意力应该花在"方向对不对"上，不是"执行对不对"上。

## 几个小但重要的工程细节

**确认关键词要精确匹配。** 最初用 `"好" in text` 判断用户确认。结果"不好意思我在忙"也被当成了确认，直接开始执行了。改成 `text.strip() == "确认"` 精确匹配。

**Workflow 状态要持久化。** 用户发了需求，AI 回了需求确认。用户过了五分钟才回"确认"——但这五分钟里 bot 可能因为某个 bug 重启了，状态全丢。所以 workflow state 要写到文件里，重启后能恢复。

**相同任务要去重。** 用户可能重复发同一个需求（网络问题、没看到回复）。每次执行前先检查历史 PRD 文件，有完成的相同任务直接返回结果。

**LLM 输出格式不可信。** 明确说了"只输出 JSON"，Claude 还是会加上 prose 说明、markdown 代码块包裹。必须用正则从整段文本里提取 JSON 对象，不能假设整个 response 就是纯 JSON。

## 回头看

整个项目从 0 到可用大概花了 6 小时。最终 11 个文件约 2000 行 Python，支持飞书 + 微信双通道。

如果让我重来，有几件事会做不同：

1. **一开始就用 stream-json 持久进程。** 我先做了 per-message spawn 版本，调通了再换持久进程，等于绕了一圈弯路。
2. **先做 single mode 跑通一切，再加 team mode。** 多 Agent 团队模式的复杂度是指数级的——调度、超时、状态机、确认流程、测试生成……一个个加上去调试成本巨大。
3. **日常用 Sonnet，Opus 只在审查/规划时用。** Opus 质量高但太慢，团队模式 20 次调用全用 Opus 要等好几分钟。
4. **测试从第一天就写。** 我是代码堆到 2000 行才开始拆分和补测试的，早期加测试后面能省很多排查时间。

## 当前状态

- 日常通勤时用飞书/微信发简单任务（改 bug、查代码、跑命令），2-5 秒回复
- 复杂任务用团队模式，3-10 分钟自动交付，手机上只需要确认两次
- Watchdog 保活 + 自动重启，基本不需要手动干预
- 成本：Claude Max 订阅（日常单次对话不额外计费）+ DeepSeek API（一天不到一块钱）
