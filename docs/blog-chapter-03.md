# 两个 AI，一个当军师一个当打手

简单任务秒回之后，人就会开始贪心。

有一天我在地铁上发了句："帮我把 ws_bot.py 拆成多个模块。"

Claude 回了我一大段分析——文件 2000 多行，涉及配置、状态管理、日志解析、CLI 调用、事件处理……它说得都对，但就是没动手。因为这事太大了，一条消息搞不定。

这就像你跟同事说"帮我重构一下这个系统"——如果他真一声不吭就开始改，你反而要慌了。正常的流程应该是：先理解需求 → 出方案 → 你确认 → 动手 → 测试 → 审查。少了任何一步，大概率翻车。

我就想：能不能让 AI 自己走完这套流程？

---

## 两个模型，两种性格

一开始我试过全部用 Claude Opus。它聪明、谨慎、理解力强——但太慢了。一个 5 步计划，每步调一次 Opus，光等回复就好几分钟。而且贵。

后来我发现 DeepSeek V4 Pro 写代码特别快，3-5 秒就能出一个完整模块的方案，质量也不错，关键是成本几乎可以忽略。但它有个问题：**只看你给它的 prompt，不了解项目全貌**。它不会主动去读文件、查依赖，写出来的 import 路径经常是编的。

这两个模型的特点刚好互补：

- **Claude Opus** — 理解力强、会读项目文件、能自主验证，但慢且贵。适合**动脑子**的活：分析需求、做决策、审查结果。
- **DeepSeek V4 Pro** — 写代码快、便宜、质量不错，但只看 prompt 不看全局。适合**动手**的活：根据明确的指令生成代码方案。

于是我给它们分了工：

| 角色 | 模型 | 职责 |
|------|------|------|
| Dispatcher（军师） | Claude Opus | 分析需求、制定计划、最终审查 |
| Executor（打手） | DeepSeek V4 | 根据计划生成代码方案 |
| Runner（执行者） | Claude Code CLI | 拿着方案在真实项目里执行、验证、修正 |

---

## 消息怎么路由

不是每条消息都需要出动"团队"。查个状态、问个问题，直接让 Claude 回答就行。只有复杂任务才需要多 Agent 协作。

我写了一个 `route_message` 函数做路由判断，优先用关键词匹配：

```python
# 关键词命中 → 直接判定模式
if any(kw in text for kw in ["拆分", "重构", "新功能", "架构"]):
    return "team"

# 简单问题 → 单次对话
if any(kw in text for kw in ["查", "看下", "什么意思", "状态"]):
    return "single"
```

90% 的消息通过关键词就能判断，不需要调 LLM 做路由。只有真的模棱两可的才 fallback 到 Claude 做语义判断。这样既快又省钱。

## Workflow 状态机

复杂任务进入团队模式后，`multi_agent.py` 里的 workflow 状态机开始运转——你可以把它想象成外卖订单的状态流转：下单 → 商家接单 → 骑手取餐 → 配送中 → 已送达。每个任务也有类似的阶段：

```
分析需求 → 制定计划 → 等待确认 → 逐步执行 → 验证 → 完成
```

每个阶段对应一个 phase 状态：

```python
workflow = {
    "phase": "executing",      # 当前阶段
    "started_at": 1716000000,  # 开始时间
    "plan": [...],             # 执行计划
    "current_step": 2,         # 当前执行到第几步
}
```

状态会持久化到 JSON 文件里——因为用户可能发了需求后过五分钟才回"确认"，这期间 bot 可能重启过。状态不能丢。

## 协作的真实流程

拿"拆分 ws_bot.py"这个任务举例，实际执行流程是：

1. **Dispatcher（Claude Opus）** 分析需求：理解要拆什么、涉及哪些依赖、有什么风险
2. **Dispatcher** 发给我确认："我理解你要将 ws_bot.py 按职责拆分为 config/state/text_utils 等模块，涉及 8 个文件，确认？"
3. 我回"确认"
4. **DeepSeek** 针对每一步生成代码方案（比如 config.py 该包含哪些内容）
5. **Claude Code** 拿着方案在真实项目里执行——它能读现有文件、看 import 关系，所以会自动修正 DeepSeek 方案里不对的地方（比如 import 路径不存在的）
6. 每步执行完跑验证
7. 全部完成后 **Dispatcher** 做最终审查

关键是第 5 步：DeepSeek 出的方案有时候 import 路径是它编的（它又没读过你的项目）。但 Claude Code 在真实环境里执行的时候会发现"这个模块不存在"，然后自动修正。两个模型的能力互补在这里体现得最明显。

## TDD：先写测试再写代码

团队模式一开始有个严重的问题：代码写完了不知道对不对。要到最后审查才发现搞错了，然后整个推倒重来。

后来我加了 TDD（测试驱动开发）——简单说就是"先出考题再答卷"。Dispatcher 制定计划的时候，每一步除了描述"做什么"，还要先写一个 **验收断言**（考题）。

比如"提取 config.py"这一步的验收标准是：

```python
assert Path('config.py').exists()
assert 'Settings' in open('config.py').read_text()
```

执行流程变成了：

1. 根据验收标准先生成测试脚本（此时代码还不存在）
2. DeepSeek 出方案
3. Claude Code 执行
4. 跑预生成的测试
5. 通过 → 继续；失败 → 修一轮

好处是执行者**写代码的时候已经知道什么算完成**。它不会跑偏，不会过度设计。而且如果 DeepSeek 的方案有问题（比如少了某个函数），测试会立刻暴露出来，不用等到最后审查。

## 为什么不全用一个模型

经常有人问：Claude Opus 这么强，为什么不让它一个人全干了？

三个原因：

1. **慢。** Opus 做一步要 10-30 秒。5 步计划全走 Opus 要等好几分钟。DeepSeek 3-5 秒就出方案。
2. **贵。** Opus 的 token 价格是 DeepSeek 的几十倍。团队模式一个任务可能调 20 次模型，全走 Opus 成本不可控。
3. **互补。** Opus 擅长理解和决策，DeepSeek 擅长生成代码。让各自做各自擅长的事，总体效果最好。

反过来也一样——不能全用 DeepSeek。它不了解项目全貌，做不了"这个需求到底该改哪些文件"这种决策，也不会主动去验证代码能不能跑通。

顺便说一下：有人问我为什么不用 LangChain 这类现成框架。原因很简单——我的场景是 CLI 子进程 + 飞书消息队列，LangChain 的抽象层对不上。与其花时间绕过框架的限制，不如直接写 300 行状态机，全部逻辑一目了然。做架构选型，"我能完全理解并掌控"比"功能列表很长"重要得多。

架构设计得挺漂亮。我甚至有点得意。

然后上线第二天，我随手发了句"帮我加个用户认证"——十五分钟后出站一看，项目代码被改得面目全非，新建了三个目录，加了两个我不需要的依赖。方向完全错误。
