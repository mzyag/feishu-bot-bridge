# 地铁上的一个念头

13 号线，早高峰，我单手扶着杆，另一只手举着手机刷消息。脑子里突然冒出来一个想法：昨晚那段代码有个边界条件没处理。

然后呢？然后就没有然后了。等我到公司，打开电脑，那个想法早不知道飘哪去了。

这种事发生过太多次。通勤一个小时，脑子里经常蹦出"对了，那个函数要加个判断"之类的想法，但它们的寿命大概只有三站地铁。我试过往备忘录里记，但到公司看备忘录的时候写的是"改那个东西"——什么东西？哪个？过去的我对现在的我毫无交代。

有段时间我试过在手机上 SSH 到服务器。怎么说呢——如果你在晃动的车厢里用虚拟键盘敲过 `kubectl get pods --namespace=production`，你就知道那不是人类应该做的事。没有 Tab 补全，一个 typo 就要退格重来，手指按错一行命令就可能干出危险的事。而且地铁一进隧道信号就没了，SSH 断了重连又得登录。

我当时就想：为什么不能像跟同事说话一样？"帮我看下那个函数有没有处理空字符串的情况"——说完就完事了，不用打命令，不用开电脑。

飞书和微信我天天都开着。如果能在聊天窗口里直接跟一个 AI 工具对话，让它帮我读代码、改代码、跑测试……那我在地铁上就能干活了。

---

## 第一版：subprocess.Popen

那天晚上到家我就开始搞。思路很简单：

1. 飞书机器人收到我发的消息
2. 在 Mac 上启动 Claude Code CLI，把消息传给它
3. 等它执行完，把输出发回飞书

飞书这边好办——`lark-oapi` 有 WebSocket 长连接模式，不需要公网服务器，本地跑个 Python 脚本就能收消息。真正的问题是怎么调 Claude Code。

我用了最直觉的方式：`subprocess.Popen`。

```python
proc = subprocess.Popen(
    ["claude", "-p", user_message],
    stdout=subprocess.PIPE,
    stderr=subprocess.PIPE,
    text=True,
)
stdout, _ = proc.communicate(timeout=120)
```

每收到一条飞书消息，就 spawn 一个 Claude Code 子进程，让它处理完，拿 stdout 的内容发回去。简单粗暴。

第一条消息发过去的时候我盯着终端，心跳加速。三秒……五秒……叮——飞书收到了回复。"你好！有什么我可以帮你的？"

能跑了！我兴奋了大概十分钟。

然后就开始难受了。

## 5-10 秒的痛

问题是**慢**。不是网络慢，是 Claude Code 每次启动都慢。

它启动的时候要加载 MCP servers、扫描项目里的 CLAUDE.md 文件、初始化各种上下文……在我那台 Mac 上，每次冷启动大概 5-10 秒。也就是说每发一条消息，都要等至少 5 秒才有反应。

5 秒听起来不多，但在聊天场景里是致命的。你发了个问题，盯着屏幕等……5 秒……8 秒……你开始怀疑它是不是挂了。到第 10 秒你已经在想"算了我自己来吧"。

更糟的是没有上下文。每条消息都是独立的子进程，彼此不知道对方的存在。我问"帮我看下 config.py 的第 30 行"，然后紧接着问"那个变量是什么意思"——它不知道"那个"指什么，因为上一个进程已经死了。就像每次跟人说话都要重新自我介绍。

我在终端里看着 `top`，每收到一条消息 CPU 就飚一次然后掉下来，像心电图。启动、执行、销毁，启动、执行、销毁。笨得要命。

## 发现 stream-json

第二天通勤的时候我又在刷 Claude Code 的文档（对，在地铁上），翻到了一个参数：

```
--input-format stream-json
--output-format stream-json
```

这是一个我之前完全忽略的运行模式。简单说：

- 启动 Claude Code 的时候带上这个参数，它就**不会退出**
- 你往它的 **stdin** 写 JSON 格式的消息，它就处理
- 它通过 **stdout** 持续输出 JSON 格式的事件流——处理进度、工具调用、最终结果都在里面
- 关键是：**上下文在进程内部自动保持**。进程活多久，记忆就在多久

这意味着我不需要每条消息都启动新进程了。我只要启动**一个** Claude Code 进程，保持它不退出，所有消息都往这一个进程里发就行。

```python
proc = subprocess.Popen(
    ["claude", "-p", "--input-format", "stream-json",
     "--output-format", "stream-json"],
    stdin=subprocess.PIPE,
    stdout=subprocess.PIPE,
    text=True,
    bufsize=1,
)
```

发消息变成了往 stdin 写一行 JSON：

```python
msg = {"type": "user", "message": {"role": "user", "content": [{"type": "text", "text": user_input}]}}
proc.stdin.write(json.dumps(msg) + "\n")
proc.stdin.flush()
```

收回复变成了读 stdout 的事件流：

```python
for line in proc.stdout:
    event = json.loads(line.strip())
    if event.get("type") == "result":
        reply = event.get("result", "")
        break
```

就这么简单。一个进程常驻，所有对话都走这一个管道。

## 质变

切到 stream-json 模式之后，响应时间从 5-10 秒直接降到了 **2 秒以内**。

那 2 秒纯粹是模型生成第一个 token 的延迟——进程本身零启动开销，因为它一直活着。上下文完全连续：我问"帮我看下 config.py"，紧接着问"第 30 行那个变量是干嘛的"，它知道我在说什么。不用重复交代背景，不用每次把聊天历史重新喂给它。

体感就是"消息发出去，手机还没放下就回了"。

我连发了五条相关的问题测试上下文——每一条都精准承接了上文的内容。就像跟一个真的在听你说话的人聊天，而不是一条金鱼。

那个晚上我在飞书群里发了句"帮我查下 ws_bot.py 里有没有未处理的异常"，2 秒后收到了一份详细的分析。我喝了口凉透的茶，觉得这东西真的能用了。

---

2 秒响应，上下文连续，完美——直到我关掉终端。程序跟着就没了。

我需要让它自己活着。
