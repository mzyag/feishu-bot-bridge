# 如果重来一次

两周后的一个早上，我突然意识到自己已经很久没手动重启过 bot 了。它就那么安静地跑在 Mac mini 后台，每天跟着我通勤，等着飞书或微信里的消息。

回头算了一下：从第一行代码到稳定可用，大概花了 6 个小时。11 个 Python 文件，2000 行左右。听起来不多，但中间的弯路够写一本避坑指南。

---

## 四个"我不会再这么做"

### 1. 持久进程应该从第一天就用

我一开始做了 per-message spawn 版本——每条消息起一个新进程。调通了，能跑了，我就觉得"先这样吧"。

结果用了两天之后才去换持久进程。那两天的体验差极了（每次 5-10 秒等待），但因为"能用"，我就不想动了。"能跑"是优化的天敌。

如果重来，我会直接上 stream-json 持久进程，跳过 spawn 版本。反正最终一定要换的，不如一开始就做对的方案。

### 2. 先稳定再加复杂度

单消息模式跑通之后我立刻开始做多 Agent 团队模式。结果两边同时有 bug：基础的消息处理有时候丢消息，团队模式的状态机也有问题。两个系统交织在一起排查，简直噩梦。

应该先把 single mode 打磨到稳如老狗（至少跑一周没问题），再加 team mode。复杂度应该一层一层往上叠，不能两层同时叠。

### 3. 模型分级使用

一开始我所有调用都走 Claude Opus——最强最贵最慢。团队模式一个任务可能调 20 次模型，全走 Opus 要等好几分钟，成本也扛不住。

后来改成分级：
- **规划和审查** → Opus（需要深度理解）
- **日常对话和执行** → Sonnet（快且够用）
- **代码方案生成** → DeepSeek（极快极便宜）

改了之后体感速度翻倍，成本降了一个数量级。

### 4. 测试从第一天就写

我是攒到 2000 行才开始补测试的。重构的时候心惊胆战——改一个函数不知道会不会影响另外三个地方。

如果从第一天就写测试，后面加 watchdog、拆文件、改路由的时候会踏实很多。这条道理人人都知道，但写 side project 的时候谁不是"先跑通再说"呢。代价就是后面补的时候要花三倍时间。

## 现在的数据

跑了大概两周，数据稳定了：

| 场景 | 响应时间 | 说明 |
|------|---------|------|
| 简单任务（查代码、改配置、问问题） | 2-5 秒 | 直接走 Claude 持久进程 |
| 复杂任务（重构、新功能、多步操作） | 3-10 分钟 | 团队模式，手机确认两次 |

成本：
- Claude：Max 订阅，日常对话不额外计费
- DeepSeek API：一天几毛到一块钱
- 总计：**一天不到一块钱**（不算订阅的话）

可用性：
- Watchdog 保活 + 自动重启，两周没手动干预过
- 飞书 + 微信双通道，哪个顺手用哪个

## 最后

最让我满意的时刻不是代码写得多优雅，也不是架构设计多巧妙。

是某天早高峰，我在 13 号线上发了句"帮我把 ws_bot.py 拆成多个模块"。然后塞上耳机听歌。

到站的时候手机振了一下，飞书通知："重构完成，8 个文件全部测试通过。"

走进办公室，`git pull`，代码整整齐齐摆在那了。config、state、feishu_api、text_utils、log_viewer、codex_runner、claude_session——每个模块各司其职。

那种感觉就像你有了一个随叫随到的搭档，而它永远不会说"等我到工位再看"。
